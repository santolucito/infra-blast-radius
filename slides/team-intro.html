Infrastructure security tooling

Which fix is safer?

A tool that compares two infrastructure changes and tells you which one grants less access — and why.

Comparative Security Blast Radius  ·  team intro

01The question no existing tool answers

You have a security finding — an over-permissioned role, an open port. You write a fix. A teammate writes a different fix. In review, someone asks:

“These both work… but which one is safer?”
  • terraform plan tells you what changes, not how much risk it adds.
  • Security scanners rate one state at a time — they don't compare two.
  • So the call gets made by gut feel in a PR thread.

02Why terraform plan can't help

Two fixes for the same problem. To plan, they look identical:

# module.job.aws_iam_role_policy.this will be updated in-place ~ resource "aws_iam_role_policy" "this" { name = "job" ~ policy = jsonencode(...) # <- "one policy changed". same for both fixes. } Plan: 0 to add, 1 to change, 0 to destroy.

"One IAM policy changed" is true of a one-line scope-down and of granting admin. The change is the same size. The blast radius is not.

03"Blast radius" — the security sense

How much a change expands what an attacker could reach.

Identity reach
which (action → resource) a principal can now do — expanding wildcards, managed policies, PassRole chains.
Network reach
which resources are reachable from untrusted origins — public IP + open security group + route.

mostly static  computable from config — no credentials, no apply. So it runs in CI and on two git branches.

04What the tool does

Point it at two git branches (baseline + a fix, or fix-A vs fix-B). It scores each branch's security blast radius, diffs them, and prints which grants less access — with the specific grants that drove the number.
git: fix-Agit: fix-B ──▶blast-compare ──▶verdict + reasons

Two frontends: a CLI (for CI / PRs) and a VS Code view. Today's demo is the CLI.

05The stance: we don't build the analysis

Security analysis is a solved, mature space. We orchestrate existing analyzers, normalize their output into one model, and own the comparison.

AnalyzerChannelWhat it sees
Cloudsplainingiam IAM action risk — priv-esc, data-exfil, perms-mgmt; wildcard/resource aware
Checkovnetwork IaC misconfig — open SGs, public buckets, missing encryption
granted-vs-usedcode static SDK-call scan — grants the app code never actually invokes

Each is optional; used when installed. New analyzers plug in behind one interface.

06How it works

baseline / fix-A / fix-B ──▶ checkout each ref
in an isolated worktree ──▶ run analyzers
normalize → one Finding model ──▶ diff each fix vs baseline ──▶ weighted score ──▶ ranked verdict
  • Findings are keyed by source + channel → tools merge without double-counting.
  • Weights live in a file — tunable to your threat model (you'll see why this matters).
The example · a real tradeoff

A batch job that's over-permissioned

Baseline — everyone agrees this must change:

iam/policy.json { "Effect": "Allow", "Action": "s3:*", "Resource": "*" }
infra/net.tf security group "job": ingress 443 from 10.0.0.0/8 # private. locked down. instance: public_ip = false

Broad IAM, but the network is tight. Two engineers each open a fix…

The two fixes · side by side

Both cut s3:* — so what does each trade?

fix-A · "tight IAM" fix-B · "keep network shut"
IAM s3:GetObject, s3:PutObject · on job-data/*
2 scoped actions		 s3:GetObject, s3:PutObject ＋
secretsmanager:GetSecretValue
dynamodb:GetItem, dynamodb:PutItem
5 scoped actions — a superset of fix-A
Network opened: SG → 0.0.0.0/0, public IP
reachable from the internet		 unchanged — stays private
locked down, as before

Neither is strictly safer. On IAM, fix-B contains fix-A and adds more — it accepts broader IAM to keep the network shut. fix-A keeps IAM minimal but opens the network. That's the whole tradeoff.

Gut check

Which looks safer?

Most reviewers pick fix-A — its IAM is tighter, and "open a security group to reach a license server" sounds routine.

Hold that instinct. Now ask the tool. →

07The verdict — default weights

$ blast-compare --base main --ref:A fix-A --ref:B fix-B analyzers: cloudsplaining, checkov A (fix-A) score 617 • network_exposure: 5 ← the open security group • public_exposure: 1 ← the public IP B (fix-B) score 180 • adds 3 scoped IAM actions ✅ Smallest blast radius: B (fix-B) — 3.4× smaller than A

The tool contradicts the eye: the prettier diff is 3.4× worse. One open security group outweighs three scoped grants.

08It's a knob, not a fact

The entire gap is the network channel. Your threat model says egress is behind a firewall? Discount it:

$ blast-compare … --weights network-discounted.json A (fix-A) score 24 B (fix-B) score 15 ✅ Smallest blast radius: B — 1.6× smaller (near tie)
Default
3.4×
Network discounted
1.6×

There is no universal winner — there's a tradeoff axis, and you set where you sit on it.

09Honest about what it can't do yet

The tool has no resource sensitivity. It doesn't know that fix-B's secret is the crown-jewel license key, or whether fix-A's open port reaches anything sensitive at all.

For the deepest question — "is holding this secret worse than opening that port?" — it makes the tradeoff legible and hands you a tunable knob. It's decision support, not a decision oracle.

Resource-sensitivity weighting (secrets/KMS/prod-DB = high) is designed but not built. That's the piece that would settle the hardest tradeoffs.

10Why this is new

ToolCan it say "fix-B is safer than fix-A, here's why"?
terraform planNo — both are "1 policy changed"
grep / action countNo — same action set; the signal is scoping + channel
Cloudsplaining aloneNo — blind to the network hole
Checkov aloneNo — blind to IAM reach; doesn't rank two branches
blast-compareYes — normalized, cross-channel, A/B verdict + drivers

The moat isn't analysis depth — it's the comparison, the normalization across tools, and the tunable score.
